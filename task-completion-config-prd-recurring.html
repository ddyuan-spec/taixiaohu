PART 1 基础功能说明

1.1 功能概述

泰小虎客户端已规划"任务体系"（新手引导、每日任务等），每个任务在客户端有明确的完成判定逻辑。以"完善健康档案"为例，健康档案为 6 步引导式（性别 / 生日 / 身高 / 体重BMI / 睡眠 / 上传体检报告），其中上传体检报告为非必填末步——只要前面 5 个必填步骤填了即视为建档完成：

  • 情形 1：用户完成前面必填步骤（体检报告可传可不传）→ 已完成
  • 情形 2：用户中途跳过、零数据 → 未完成
  • 情形 3：用户中途跳过但填了部分必填步骤 → 部分完成，但需保留已填数据并在后续引导中带入默认值补全

问题：这些完成逻辑散落在客户端代码中，运营无法自助配置，且不同任务判定标准差异大，缺少统一抽象。

目标：在后台管理（任务配置模块）提供一套通用、可配置的"完成条件"配置能力，供运营配置 / 编辑已有任务的完成判定、状态流转与奖励发放。⚠️ 任务不由运营手动新增——任务均由研发通过代码 / 部署创建，后台运营仅可对已有任务进行配置与编辑，不可新建、不可删除。本 PRD 同时约定：每个任务的判定必须落地为埋点事件（见各任务的「埋点信息」章节，格式对齐《泰小虎埋点表 v2.3》现版）。

1.3 任务类型（一次性 / 周期重复）

每个任务必须指定「任务类型」，两类任务的生命周期与奖励模型完全不同：

维度一次性任务 one_time周期重复任务 recurring
完成后永久"已完成"，不再重置到达周期重置点后回到"未完成"，可再次完成、再次领奖
奖励模型两态固定奖励（完成 +N，部分完成 +0；「未完成」固定 0 不可配置）按虎力值等级差异化发放（见 1.4），两态固定奖励不生效
周期配置无必填：周期单位（每日/每周/每月）+ 可配置重置时间
典型任务完善健康档案、创建打卡计划、完成首次打卡、发起首轮对话浏览健康好物（每日）、每日打卡（每日）
字段差异：选择「周期重复」后，编辑弹窗展开「周期配置区」（周期单位 + 重置时间 + 虎力值等级奖励表），并隐藏两态固定奖励输入框；选择「一次性」则相反。后台按任务类型动态渲染表单，避免无关字段干扰运营。

1.4.2 虎力值等级奖励模型

泰小虎虎力值体系存在用户等级概念。周期任务的奖励随用户当前等级差异化发放：后台为周期任务配置若干「等级区间 → 单次奖励 + 周期内累计上限」的规则行。判定发奖时，服务端读取该用户当前虎力值等级，命中对应区间，按该区间的单次奖励发放；当本周期内该用户在该任务上累计发放已达上限，则任务置为"已完成"并不再发放，直至下一周期重置或用户升级到更高区间。

1.5 任务两态模型（已完成 / 部分完成）

采用显式两态模型（已完成 / 部分完成），而非简单的是/否：

状态含义是否发虎力值客户端展示
部分完成命中"部分完成"条件组（如跳过但填了部分必填步）否（仅完成发）进度 + "继续完善"按钮（绿）
已完成命中"已完成"条件组（如建档完整度达 100%）是（发全额）"领取虎力值"（金）→ "已完成"灰
状态流转：未完成 → 部分完成 → 已完成 单向升级；一旦"已完成"且领取后不可逆。优先级解析：已完成 > 部分完成 > 未完成。注意：「未完成」固定 0 虎力值且不可在后台配置，仅「部分完成」「已完成」两态可配置虎力值。

1.6 任务引擎配置模型

每个任务的"完成条件"由以下 6 个部分构成：

序号配置项说明控件
①判定类型当前统一为"任务判定"（灰显固定，不可改）文本/禁用
②判定属性（写死）各状态判定属性写死，以中文名显示、不可编辑；用户仅可改关系符与值、不可增删条件文本/只读
③属性条件组属性写死、不可增删；仅关系符与值可改，多行 AND 组合固定行
④状态映射将命中的条件组映射到 已完成 / 部分完成（两态）映射表
⑤续填策略开关 + 默认值来源（用于部分完成态的补全）开关
⑥奖励分配仅「部分完成」「已完成」两态可配置虎力值；「未完成」固定为 0 且不可配置数字输入（部分完成 / 已完成）
设计取舍：底层判定仍依赖埋点上报属性（如 archive_result），但对运营屏蔽事件 Key，改由任务类型写死「判定属性」并以中文名呈现；属性不可编辑、不可增删条件，用户仅调关系符与值，模型更可控。

1.7 属性条件组 · 配置规范

判定属性由任务类型写死（以中文名显示、不可编辑），用户仅可改关系符与值、不可增删条件。每组条件由若干行组成，行内为：

[属性名]  [运算符]  [值]  （同一组内多行 = AND）

元素可选值示例
属性名（写死）由任务类型写死的判定属性（如「状态」「打卡次数」「页面」），不可编辑status / count / page（中文名呈现）
运算符等于 / 不等于 / 大于等于 / 小于等于 / 包含 / 非空存在== / >= / 非空
值字符串 / 数字 / 枚举first_onboarding / 1 / service
  • 条件组 A → 已完成：如 complete_source 非空（即 profile_complete 触发）
  • 条件组 B → 部分完成：如 is_onboarding_end == true AND archive_result == partial
  • 均不命中 → 保持未完成

1.8 实时判定引擎

采用 实时事件驱动：埋点上报即刻判定，用户领奖无延迟。流程如下：

  1. 客户端 / 服务端上报埋点事件，必须携带 user_id 及判定所需属性（如 complete_source、archive_result、user_checkin_count）。
  2. 后端事件流实时消费，取出该用户所有处于"未完成 / 部分完成"状态的任务配置。
  3. 对每个任务，依次用其"属性条件组"匹配当前事件：命中"已完成"组 → 置已完成并发奖；命中"部分完成"组 → 置部分完成（不发奖）；均不中 → 保持原态。
  4. 状态变更写回"用户任务进度表"，推送客户端刷新任务卡。
  5. 已完成态幂等：同一事件重复上报不重复发奖；部分完成可再次上报升级到已完成。
  6. 周期重置（仅周期重复任务）：后台定时任务在配置的重置点，把该任务已完成的用户进度重置为"未完成"、清零"当前周期已发放"，用户进入新周期可再次完成领奖。

1.9 客户端协同

7.1 两态展示

  • 部分完成：卡片显示进度 + "继续完善"按钮（绿），点击进入带默认值的引导流。
  • 已完成（未领）：显示"领取虎力值"（金）→ 点击弹气泡、按钮变"已完成"灰不可点。

7.2 续填机制（以完善健康档案为例）

续填本质是客户端行为，后台仅下发开关 + 默认值配置。用户处于"部分完成"后再次进入建档引导：

preFill = 用户已填字段（profile_fields） ∪ 默认值补全项

客户端读取任务配置，进入引导流时预填已填字段、其余用默认值；补完点完成 → 档案完整度达 100% → 上报 profile_complete（complete_source=update_fill）→ 任务升级为已完成、发奖。补填动作本身亦由 profile_update（update_source=memory_fill）记录。

1.10 埋点规范约束

  • 事件必须携带 user_id，否则无法关联用户任务进度。
  • 事件属性需包含判定所需字段（complete_source / archive_result / user_checkin_count / recommend_source 等），缺失属性按"不命中"处理。
  • 实时上报（非批量延迟），保证判定时效。
  • 事件定义须登记于《泰小虎埋点表 v2.3》现版；已有事件直接复用，缺口事件/属性（如 plan_create、checkin_record、profile_submit.is_onboarding_end/archive_result、recommend_source 补 service）须先补齐方可驱动经验。

1.11 边界与异常处理

异常场景处理策略
事件缺失判定属性按"不命中"处理，任务保持原态
同一事件多次上报已完成态幂等，不重复发奖
条件组为空该任务不可经事件完成（需人工 / 灰度兜底）
多条件组同时命中按优先级 已完成 > 部分完成 解析
