PART 1 基础功能说明

1.1 功能概述

泰小虎客户端已规划"任务体系"（新手引导、每日任务等），每个任务在客户端有明确的完成判定逻辑。以"完善健康档案"为例，健康档案为 6 步引导式（性别 / 生日 / 身高 / 体重BMI / 睡眠 / 上传体检报告），其中上传体检报告为非必填末步——只要前面 5 个必填步骤填了即视为建档完成：

  • 情形 1：用户完成前面必填步骤（体检报告可传可不传）→ 已完成
  • 情形 2：用户中途跳过、零数据 → 未完成
  • 情形 3：用户中途跳过但填了部分必填步骤 → 部分完成，但需保留已填数据并在后续引导中带入默认值补全

问题：这些完成逻辑散落在客户端代码中，运营无法自助配置，且不同任务判定标准差异大，缺少统一抽象。

目标：在后台管理（任务配置模块）提供一套通用、可配置的"完成条件"配置能力，供运营查看已有任务的完成判定、状态流转与奖励发放，并对任务做启用 / 停用。⚠️ 任务不由运营手动新增 / 编辑 / 删除——任务均由研发通过代码 / 部署创建，后台运营仅可查看 + 启用 / 停用，不可新建、不可编辑、不可删除。本 PRD 同时约定：每个任务的完成判定以「后端事件中文名」表达（如健康档案的「性别 / 出生日期 / 身高 / 体重 / 睡眠」字段非空），不再使用埋点事件 Key；各任务对应的跳转目标以中文页名登记于「页面跳转」列（见 1.2 与 PART 2 任务总览表）。

1.3 任务类型（一次性 / 周期重复）

每个任务必须指定「任务类型」，两类任务的生命周期与奖励模型完全不同：

维度一次性任务 one_time周期重复任务 recurring
完成后永久"已完成"，不再重置到达周期重置点后回到"未完成"，可再次完成、再次领奖
奖励模型两态固定奖励（完成 +N，部分完成 +0；「未完成」固定 0 不可配置）按虎力值等级差异化发放（见 1.4），两态固定奖励不生效
周期配置无必填：周期单位（每日/每周/每月）+ 可配置重置时间
典型任务完善健康档案、创建打卡计划、完成首次打卡、发起首轮对话浏览健康好物（每日）、每日打卡（每日）
字段差异：选择「周期重复」后，编辑弹窗展开「周期配置区」（周期单位 + 重置时间 + 虎力值等级奖励表），并隐藏两态固定奖励输入框；选择「一次性」则相反。后台按任务类型动态渲染表单，避免无关字段干扰运营。

1.4.2 虎力值等级奖励模型

泰小虎虎力值体系存在用户等级概念。周期任务的奖励随用户当前等级差异化发放：后台为周期任务配置若干「等级区间 → 单次奖励」的规则行。判定发奖时，服务端读取该用户当前虎力值等级，命中对应区间，按该区间的单次奖励发放；每日完成次数达「每日可完成次数」上限后停发，直至次日 00:00 重置或用户升级到更高区间。

1.5 任务两态模型（已完成 / 部分完成）

采用显式两态模型（已完成 / 部分完成），而非简单的是/否：

状态含义是否发虎力值客户端展示
部分完成命中"部分完成"条件组（如跳过但填了部分必填步）否（仅完成发）进度 + "继续完善"按钮（绿）
已完成命中"已完成"条件组（如建档完整度达 100%）是（发全额）直接发放虎力值（金），无"领取"中间态 → 置"已完成"灰
状态流转：未完成 → 部分完成 → 已完成 单向升级；任务命中"已完成"即直接发放虎力值，无"领取"中间态，发放后不可逆。优先级解析：已完成 > 部分完成 > 未完成。注意：「未完成」固定 0 虎力值且不可在后台配置，仅「部分完成」「已完成」两态可配置虎力值。

1.6 任务引擎配置模型

每个任务的"完成条件"由以下 6 个部分构成：

序号配置项说明控件
①判定类型当前统一为"任务判定"（灰显固定，不可改）文本/禁用
②判定属性（写死）各状态判定属性写死，以中文名显示、不可编辑；用户仅可改关系符与值、不可增删条件文本/只读
③属性条件组属性写死、不可增删；仅关系符与值可改，多行 AND 组合固定行

1.8 实时判定引擎

采用 实时事件驱动：埋点上报即刻判定，用户领奖无延迟。流程如下：

  1. 客户端 / 服务端上报埋点事件，必须携带 user_id 及判定所需属性（如 complete_source、completion_rate、user_checkin_count）。
  2. 后端事件流实时消费，取出该用户所有处于"未完成 / 部分完成"状态的任务配置。
  3. 对每个任务，依次用其"属性条件组"匹配当前事件：命中"已完成"组 → 置已完成并发奖；命中"部分完成"组 → 置部分完成（不发奖）；均不中 → 保持原态。
  4. 状态变更写回"用户任务进度表"，推送客户端刷新任务卡。
  5. 已完成态幂等：同一事件重复上报不重复发奖；部分完成可再次上报升级到已完成。
  6. 周期重置（仅周期重复任务）：后台定时任务在配置的重置点，把该任务已完成的用户进度重置为"未完成"、清零"当前周期已发放"，用户进入新周期可再次完成领奖。

1.9 客户端协同

7.1 两态展示

  • 部分完成：卡片显示进度 + "继续完善"按钮（绿），点击进入带默认值的引导流。
  • 已完成：任务完成瞬间直接发放虎力值（弹气泡 +N），卡片置"已完成"灰不可点，无需"领取"操作；同时写入「虎力值变动记录」。

7.2 续填机制（以完善健康档案为例）

续填本质是客户端行为，后台仅下发开关 + 默认值配置。用户处于"部分完成"后再次进入建档引导：

preFill = 用户已填字段（profile_fields） ∪ 默认值补全项

客户端读取任务配置，进入引导流时预填已填字段、其余用默认值；补完点完成 → 档案完整度达 100% → 上报 profile_complete（complete_source=update_fill）→ 任务升级为已完成、发奖。补填动作本身亦由 profile_update（update_source=memory_fill）记录。

1.10 埋点规范约束

  • 事件必须携带 user_id，否则无法关联用户任务进度。
  • 事件属性需包含判定所需字段（complete_source / completion_rate / user_checkin_count / recommend_source 等），缺失属性按"不命中"处理。
  • 实时上报（非批量延迟），保证判定时效。
  • 事件定义须登记于《泰小虎埋点表 v2.3》现版；已有事件直接复用，缺口事件/属性（如 plan_create、checkin_record 两事件需新增；recommend_source 需补充 service 枚举）。注：profile_submit.is_onboarding_end / archive_result 非真实埋点属性（v2.3 无此规划），非缺口，原 PRD 误引，已移除。

1.11 边界与异常处理

异常场景处理策略
事件缺失判定属性按"不命中"处理，任务保持原态
同一事件多次上报已完成态幂等，不重复发奖
条件组为空该任务不可经事件完成（需人工 / 灰度兜底）

1.13 上线前已完成任务的虎力值补发

用户在功能上线前若已满足某些任务的完成条件（如健康档案早已填好、早就发起过对话），首次进入任务体系时不应"错过"这部分虎力值。

触发时机：客户端拉取任务列表时，由后端统一做一次"历史完成回溯"——对当前用户已满足完成条件但尚未发放虎力值的任务，自动补齐发放并写入变动记录（变动原因标记为"上线前补发"）。

  • 补发仅针对"条件已满足但未发奖"的任务，已发放过的任务幂等不重复发。
  • 补发与日常实时判定共用同一套"已完成"判定逻辑（见 1.8），不引入额外分支。
  • 前端在拉取任务列表的回调里刷新虎力值总额与变动记录，无需单独交互。

1.14 每日任务说明（刷新 / 接近 0 点 / 每日可完成次数 / 周期上限）

「浏览健康好物」「分享泰小虎」为每日任务，每天可完成「每日可完成次数」次（默认 1，仅限正整数，不含 0）并领奖，次日重置。关键逻辑：

逻辑点说明
每日刷新时间后台可配置重置时间（默认每日 00:00）。到点将该任务处于"已完成"的用户进度重置为"未完成"，并清零"当日已发放"，用户即可在新的一天重新完成领奖。
接近 0 点完成若用户在 23:59 完成并领奖，按"当日"计入；跨过 00:00 重置后即进入次日，需重新完成。重置以服务端时间为准，避免客户端时间偏差导致重复 / 漏发。
每日可完成次数同一用户同一每日任务，自然日内最多可完成并领奖「每日可完成次数」次（默认 1，仅限正整数，不含 0）；达到次数上限后重复触发（如多次进入健康好物页 / 多次分享）不重复发奖（幂等）。
每日任务的「重置时间 / 每日奖励 / 每日可完成次数」均为后台可配置项（见各任务数据设置与平台端原型），由研发在后台配置，运营不可改任务本身、仅可启停。
