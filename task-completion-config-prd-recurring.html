PART 1 基础功能说明

1.1 功能概述

泰小虎客户端已规划"任务体系"（新手引导、每日任务等），每个任务在客户端有明确的完成判定逻辑。以"完善健康档案"为例，健康档案为 6 步引导式（性别 / 生日 / 身高 / 体重BMI / 睡眠 / 上传体检报告），其中上传体检报告为非必填末步——只要前面 5 个必填步骤填了即视为建档完成：

  • 情形 1：用户完成前面必填步骤（体检报告可传可不传）→ 已完成
  • 情形 2：用户中途跳过、零数据 → 未完成
  • 情形 3：用户中途跳过但填了部分必填步骤 → 部分完成，但需保留已填数据并在后续引导中带入默认值补全

问题：这些完成逻辑散落在客户端代码中，运营无法自助配置，且不同任务判定标准差异大，缺少统一抽象。

目标：在后台管理（任务配置模块）提供一套通用、可配置的"完成条件"定义能力，让运营无需改代码即可定义/调整任务的完成判定、状态流转与奖励发放。本 PRD 同时约定：每个任务的判定必须落地为埋点事件（见各任务的「埋点信息」章节，格式对齐《泰小虎埋点表 v2.3》现版）。

1.3 任务类型（一次性 / 周期重复）v2.2 新增

v2.2 起，每个任务必须指定「任务类型」，两类任务的生命周期与奖励模型完全不同：

维度一次性任务 one_time周期重复任务 recurring
完成后永久"已完成"，不再重置到达周期重置点后回到"未完成"，可再次完成、再次领奖
奖励模型三态固定奖励（完成 +N，部分/未完成 +0）按虎力值等级差异化发放（见 1.4），三态固定奖励不生效
周期配置无必填：周期单位（每日/每周/每月）+ 可配置重置时间
典型任务完善健康档案、创建打卡计划、完成首次打卡、发起首轮对话浏览健康好物（每日）、每日打卡（每日）
字段差异：选择「周期重复」后，编辑弹窗展开「周期配置区」（周期单位 + 重置时间 + 虎力值等级奖励表），并隐藏三态固定奖励输入框；选择「一次性」则相反。后台按任务类型动态渲染表单，避免无关字段干扰运营。

1.4.2 虎力值等级奖励模型

泰小虎虎力值体系存在用户等级概念。周期任务的奖励随用户当前等级差异化发放：后台为周期任务配置若干「等级区间 → 单次奖励 + 周期内累计上限」的规则行。判定发奖时，服务端读取该用户当前虎力值等级，命中对应区间，按该区间的单次奖励发放；当本周期内该用户在该任务上累计发放已达上限，则任务置为"已完成"并不再发放，直至下一周期重置或用户升级到更高区间。

字段说明示例
起始等级（含）该规则行生效的最低等级Lv1
结束等级（含）该规则行生效的最高等级Lv3
单次奖励（虎力值）该等级区间用户单次完成发放的虎力值5
周期累计上限（虎力值）该等级区间用户在单个周期内最多可获得的虎力值；达上限置"已完成"停发50

1.5 任务三态模型

采用显式三态模型，而非简单的是/否：

状态含义是否发虎力值客户端展示
未完成未触发任何完成/部分条件否"去完成"按钮（橙）
部分完成命中"部分完成"条件组（如跳过但填了部分必填步）否（仅完成发）进度 + "继续完善"按钮（绿）
已完成命中"已完成"条件组（如建档完整度达 100%）是（发全额）"领取虎力值"（金）→ "已完成"灰
状态流转：未完成 → 部分完成 → 已完成 单向升级；一次性任务一旦"已完成"且领取后不可逆。周期重复任务例外：到达周期重置点后由"已完成"重置回"未完成"，进入下一轮（见 1.4）。优先级解析：已完成 > 部分完成 > 未完成。

1.6 任务引擎配置模型

每个任务的"完成条件"由以下 6 个部分构成：

序号配置项说明控件
①判定类型当前统一为"埋点事件"（灰显固定，预留扩展）文本/禁用
②事件 Key监听的埋点事件名，如 profile_submit下拉 + 可输入
③属性条件组对事件属性做键值对判断，多行 AND 组合动态增删行
④状态映射将命中的条件组映射到 未完成 / 部分完成 / 已完成映射表
⑤续填策略开关 + 默认值来源（用于部分完成态的补全）开关
⑥奖励分配一次性任务：各态对应固定虎力值（部分完成通常为 0）；周期重复任务：改用「虎力值等级奖励表」（见 1.4），固定奖励不生效数字输入 / 等级奖励表
设计取舍：判定数据源统一为"埋点事件"，"字段填充型"任务的子状态编码在事件属性里（如 archive_result），无需单独一类，模型保持干净。任务类型（一次性/周期重复）只影响生命周期与奖励发放方式，不改变判定引擎本身。

1.7 属性条件组 · 配置规范

采用 键值对下拉 方式（运营友好，无需写代码）。每组条件由若干行组成，行内为：

[属性名]  [运算符]  [值]  （同一组内多行 = AND）

元素可选值示例
属性名该事件上报的属性字段complete_source / archive_result / user_checkin_count / recommend_source
运算符等于 / 不等于 / 大于等于 / 小于等于 / 包含 / 非空存在== / >= / 非空
值字符串 / 数字 / 枚举first_onboarding / 1 / service
  • 条件组 A → 已完成：如 completion_rate >= 100（即档案完整度达标）
  • 条件组 B → 部分完成：如 is_onboarding_end == true AND archive_result == partial
  • 均不命中 → 保持未完成

1.8 实时判定引擎

采用 实时事件驱动：埋点上报即刻判定，用户领奖无延迟。流程如下：

  1. 客户端 / 服务端上报埋点事件，必须携带 user_id 及判定所需属性（如 complete_source、archive_result、user_checkin_count）。
  2. 后端事件流实时消费，取出该用户所有处于"未完成 / 部分完成"状态的任务配置。
  3. 对每个任务，依次用其"属性条件组"匹配当前事件：命中"已完成"组 → 置已完成并发奖；命中"部分完成"组 → 置部分完成（不发奖）；均不中 → 保持原态。
  4. 奖励发放：一次性任务发放对应态固定虎力值；周期重复任务读取用户当前虎力值等级，命中等级奖励表对应区间，按单次奖励发放，并累加"当前周期已发放"；若达该区间周期累计上限，则置"已完成"停发。
  5. 状态变更写回"用户任务进度表"，推送客户端刷新任务卡。
  6. 已完成态幂等：同一事件重复上报不重复发奖；部分完成可再次上报升级到已完成。
  7. 周期重置（仅周期重复任务）：后台定时任务在配置的重置点，把该任务已完成的用户进度重置为"未完成"、清零"当前周期已发放"，用户进入新周期可再次完成领奖。

1.9 客户端协同

7.1 三态展示

  • 未完成：卡片显示"去完成"按钮（橙），点击跳对应页面。
  • 部分完成：卡片显示进度 + "继续完善"按钮（绿），点击进入带默认值的引导流。
  • 已完成（未领）：显示"领取虎力值"（金）→ 点击弹气泡、按钮变"已完成"灰不可点。

7.2 续填机制（以完善健康档案为例）

续填本质是客户端行为，后台仅下发开关 + 默认值配置。用户处于"部分完成"后再次进入建档引导：

preFill = 用户已填字段（profile_fields） ∪ 默认值补全项

客户端读取任务配置，进入引导流时预填已填字段、其余用默认值；补完点完成 → 档案完整度达 100% → 再次上报 profile_submit（completion_rate>=100、is_onboarding_end=true、archive_result=completed）→ 任务升级为已完成、发奖。本期完善健康档案收敛为单事件 profile_submit，不再依赖 profile_complete / profile_update 组合判定（见 2.1）。

1.10 埋点规范约束

  • 事件必须携带 user_id，否则无法关联用户任务进度。
  • 事件属性需包含判定所需字段（complete_source / archive_result / user_checkin_count / recommend_source 等），缺失属性按"不命中"处理。
  • 实时上报（非批量延迟），保证判定时效。
  • 事件定义须登记于《泰小虎埋点表 v2.3》现版；已有事件直接复用，缺口事件（如 plan_create、checkin_record）须先补齐方可驱动引擎。周期任务的等级奖励由服务端读取用户当前等级实现，不新增埋点事件。

1.11 边界与异常处理

异常场景处理策略
事件缺失判定属性按"不命中"处理，任务保持原态
同一事件多次上报已完成态幂等，不重复发奖
条件组为空该任务不可经事件完成（需人工 / 灰度兜底）
多条件组同时命中按优先级 已完成 > 部分完成 解析
部分完成 → 再次上报允许升级到已完成并发奖
周期任务当前周期已达等级累计上限置"已完成"停发，直至下一周期重置或用户升级到更高奖励区间
周期任务重置点后重复上报新周期内首次命中按新周期计数发奖，与上一周期互不影响
用户等级未覆盖任何奖励区间默认不发奖（视为无匹配规则），不阻断状态流转
周期任务未配置任何等级奖励规则配置校验拦截，无法保存（周期任务必填至少 1 条规则）
