泰小虎 · 直播挂车与时长发券

需求框架 v0.1 · 2026-07-14 · 状态：框架对齐中（关键决策已确认，细节待填充）

类型：新需求（独立分块） 关联需求：优惠券体系（调用其发券引擎） 接入：火山引擎企业直播 端：独立 H5 + 小程序 + App

1. 背景与目标

在数字人能力之前，先做独立直播 H5 页面，承载两件核心事：① 挂车——把商品加入直播间；② 观看时长发券——用户看够时长即发券，券用于小程序 / App 购物核销。直播底层是火山引擎企业直播能力，但商品卡与发券浮层由我们自研 H5 渲染，发券逻辑由优惠券体系（关联需求）提供。

2. 已确认的关键决策

决策点结论
直播接入形态自研独立 H5 + 火山直播 SDK / 拉流，自己渲染商品卡与发券浮层（不走火山原生商品卡片）
挂车商品来源商品来自「健康好物」库 → H5 内自渲染商品卡 → 点击跳 H5 落地页 → 引导小程序 / App 下单
观看时长口径火山观播 SDK 观看时长 / 心跳回调；需火山自定义登录拿 UserId 关联用户
发券逻辑归属本需求不发券、不建券，只负责"时长采集 + 阈值判断 + 调优惠券体系发券 API"（见关联需求）。H5 内不展示浮层，静默发券
用户感知方式用户登录 App / 小程序时，以弹窗 / 通知告知"获得 X 张优惠券"，点击引导至「我的优惠券」

3. 业务流程

4. 需求模块划分（5 大模块）

① 直播 H5 页（自研）

  • 独立 H5 页面框架，嵌入火山直播流
  • 整体布局（播放器 + 商品卡区 + 互动浮层）

② 挂车模块

  • 商品同步：健康好物 → 直播间商品卡
  • H5 自渲染商品卡（图 / 名 / 价 / 标签）
  • 点击跳转：H5 落地页 → 引导小程序 / App 下单

③ 观看时长采集

  • 火山观播 SDK 集成（自定义登录 UserId）
  • 观看时长 / 心跳回调采集
  • 弱网 / 退出场景的兜底计时

④ 时长发券触发

  • 阈值 / 限领判定（规则由优惠券体系管理）
  • 调用优惠券体系发券引擎 API
  • 防重复发券（每人每场限领）

⑤ 发券通知（静默后触达）

  • H5 内静默发券：不弹任何浮层/弹窗，用户无感知
  • 用户登录 App/小程序时，以弹窗/通知告知"获得 X 张优惠券"
  • 点击通知/弹窗，引导至「我的优惠券」券包列表
  • 消息中心持久化：券到账记录也可在消息中心查看

5. 核心数据模型（草案）

表关键字段说明
live_roomroom_id, title, status, start_time, end_time, volcano_room_id直播间基础信息，关联火山侧房间
watch_sessionsession_id, user_id, room_id, enter_time, exit_time, duration_sec一次观看会话，时长发券的基本单元
watch_duration_recordrecord_id, session_id, user_id, room_id,累积时长, 心跳时间戳SDK 心跳 / 兜底计时的明细，用于阈值判定
issue_trigger_loglog_id, user_id, room_id, trigger_type=live_duration, rule_id, trigger_time, coupon_result发券触发日志，记"调了优惠券体系 API + 返回"，兜底与去重用
说明：券实例（CouponInstance）、券模板、底价等数据不在本需求，由优惠券体系（关联需求）持有。本需求只记录"触发"与"心跳"，不持有券本身。

6. 发券触发设计（调优惠券体系 API）

采集 火山观播 SDK 心跳 / H5 兜底计时累计时长 → 判定 达 IssueRule 阈值（规则由优惠券体系配置，本需求读取并比对）→ 去重 按 user_id + room_id 校验本场是否已发到限领上限（参考 issue_trigger_log）→ 调用 优惠券体系发券引擎 API（传入 user_id、rule_id 等）→ 静默 H5 内不展示任何浮层，只记录本次触发结果。

本需求不实现发券规则配置、底价校验、券实例生成、用户端通知——这些分别由优惠券体系或 App/小程序 客户端完成。本需求只是"触发方"：采集时长、判定阈值、调 API。

7. 火山引擎接入要点

  • 用到的能力：① 直播流（自研 H5 通过火山直播 SDK / 拉流地址播放）；② 观播 SDK 的自定义登录（传入 UserId，用于跨端关联与发券归属）；③ 观播 SDK 的观看时长 / 心跳回调（作为发券触发依据）。
  • 本期绕过的火山能力：① 火山原生商品卡片 API（我们走 H5 自渲染挂车）；② 火山原生卡券能力（我们走自建优惠券体系）。
  • 文档参考：企业直播-商品脚本 / 观播 SDK 文档 https://docs.volcengine.com/docs/3019/1521845?lang=zh。

8. 关联需求

优惠券体系（独立需求）：本需求的"时长发券"仅是触发与展示，真正的发券引擎、券模板、发券规则（时长阈值 / 限领 / 有效期）、商品底价防击穿全部在优惠券体系实现。本需求通过发券引擎统一 API 调用，两需求解耦，互不直接改对方数据。关联点仅为"接口调用 + UserId 对齐"。

9. 待定 / 开放问题（需进一步确认）

  • 直播"一场"如何界定（同一 liveId / room_id）？用户中途退出再进，时长累计还是重置。
  • App/小程序 的"获得优惠券"通知形态：全屏弹窗一次？消息中心小红点？还是 PUSH？
  • 挂车商品卡点击后跳转的具体落地页形态，与小程序 / App 的引导链路细节。
  • 弱网 / 火山回调丢失时，H5 兜底计时的精度与去重策略。
  • 消息中心持久化：券到账记录是否需与系统消息统一存储。
