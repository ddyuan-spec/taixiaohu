泰小虎 · 优惠券体系（中台）

需求框架 v0.1 · 2026-07-14 · 状态：框架对齐中（关键决策已确认，后台范围=完整后台）

类型：新需求（独立分块） 关联需求：直播挂车/时长发券（调用本发券引擎） 端：小程序 + App + H5 归属：平台营销中台

1. 背景与目标

优惠券是平台级营销中台能力，需被多渠道复用——直播发券、任务奖励（预留）、手动运营、后续营销活动。统一建设可避免：分散发券无法统一管控底价、无法统一核销与归因。本期不绑定任何单一场景，券模板 / 规则 / 底价 / 核销 / 数据均由本需求统一管理，外部场景通过发券引擎统一 API 触发。

核心约束（用户原话）：优惠券无论如何不能导致商品超出底价（亏本）。本框架把"底价防击穿"作为发券引擎的硬校验，优先级最高。

2. 已确认的关键决策

决策点结论
定位独立优惠券中台，平台管理端统一管理：发券 / 关联商品底价 / 使用数据收集；不只服务直播
券种满减券（满 X 减 Y）+ 折扣券（打 Z 折）+ 代金券（无门槛立减） 三种
底价规则券面额 不得击穿商品底价；建券 / 发券 / 叠加 / 核销四道系统强校验
与任务体系本期不接任务引擎；发券引擎设计成可外部触发，预留接口，后续再接（券作为任务奖励）
后台范围本期做 C 完整后台：券模板配置 / 发券规则·活动配置 / 商品底价管理 / 数据看板（领取·核销·ROI）/ 用户券包查询

3. 业务流程（发券引擎被多通道调用）

4. 需求模块划分（7 大模块）

① 券模板管理

  • 券种（满减 / 折扣 / 代金）、门槛、面额、有效期、适用商品
  • 关联商品底价；模板状态（草稿 / 启用 / 停用）

② 发券引擎（统一 API）

  • 对外统一发券接口，可被多通道调用
  • 限领校验 + 底价强校验（四道关卡）
  • 生成券实例、入用户券包、触发通知

③ 用户券包

  • 用户维度持券与状态（未使用 / 已使用 / 已过期）
  • 跨端展示（小程序 / App / H5）

④ 核销

  • 下单时选券 → 校验（有效 + 适用商品 + 未超底价）
  • 计算抵扣 → 支付 → 核销实例并写记录
  • 依赖跨端 UserId 打通

⑤ 商品底价管理

  • 维护 cost_price，作为建券 / 发券 / 叠加 / 核销强校验数据源
  • 防亏本的唯一权威口径

⑥ 数据看板

  • 领取量 / 核销量 / 核销率 / ROI / 各渠道发券分布
  • 供运营评估效果

⑦ 客户端用券与入口（小程序 + App）

  • 小程序：当前「我的」页面已有「我的优惠券」入口，本期需新增券包列表页（未使用 / 已使用 / 已过期 Tab）
  • App：当前「我的」页面无优惠券入口，本期需补齐入口并同步建设券包列表页
  • 提交订单页：选择可用券、展示抵扣金额、实时校验底价（参考订单确认页优惠券行）
  • 直播发券场景：H5 内静默发券，用户登录 App/小程序后通过通知/弹窗告知收券，再引导至「我的优惠券」

6. 发券引擎设计

6.1 触发 → 发券流程

触发 直播时长达阈值（H5 静默） / 后台手动 / （预留任务） → 查规则 取对应 IssueRule → 校验限领 用户 / 本场是否已到上限 → 底价校验 券面额 ≤ 关联商品底价（叠加场景见 6.2） → 生成实例 写 CouponInstance → 通知用户 发券后通过 App/小程序 弹窗/消息中心告知用户。

底价防击穿（最高优先级，四道校验）：
  • 建券时：券面额（代金值 / 满减减额 / 折扣让利额）≤ 关联商品底价，否则禁止保存。
  • 发券时：再次校验当前底价未被调整击穿。
  • 叠加时：若用户多张可叠加券同用于一件商品，组合后抵扣总额 ≤ 商品底价；超出部分自动失效或提示。
  • 核销时：实际抵扣 = min(券面额，商品底价 − 其他已抵扣)，保证不击穿底价。

7. 核销与客户端用券流程（跨端）

用户在小程序 / App 下单 → 在提交订单页选择可用券 → 校验（券有效 + 适用商品 + 未超底价）→ 计算抵扣金额并展示在订单明细 → 支付 → 核销券实例（status=used）并写 RedeemRecord。

客户端入口：小程序「我的」已有「我的优惠券」入口；App「我的」需新增入口。券包列表页按未使用 / 已使用 / 已过期展示，支持跳转去使用/下单。跨端依赖 UserId 打通：直播 H5 通过火山自定义登录拿到 UserId，与小程序 / App 同一用户体系对齐（此打通由跨端用户中心负责，本需求消费该能力）。

提交订单页用券：参考订单确认页现有优惠券行，点击后进入选券页，列出当前订单可用券；选中后实时刷新应付总额，确保底价不被击穿。

已确认：本期做完整后台，即配置 + 数据看板全部包含

选项做哪些不做哪些运营的后果
A 接口+前台用户能看到的（券包、领券）+ 让功能跑起来的后端接口（发券 / 核销 / 底价校验）运营后台运营不能自助，需研发改代码

当前选择：C 完整后台。完整后台包含：券模板管理、发券活动 / 规则配置、商品底价管理、数据看板、用户券包查询。

9. 关联需求

① 直播挂车 / 时长发券（独立需求）：其"观看时长达阈值发券"环节，调用本需求的发券引擎统一 API。H5 内不弹浮层，静默发券；用户登录 App/小程序后通过通知/弹窗告知收券，再引导到本需求的「我的优惠券」。券模板、发券规则、商品底价均由本需求管理；直播需求只负责"时长采集 + 阈值判断 + 调 API"。

② 任务完成条件配置（独立需求，可选）：若未来任务奖励增加"发优惠券"类型，将调用本引擎。本期不实现，但发券引擎已预留外部触发接口。

10. 待定 / 开放问题（需进一步确认）

  • 券是否可叠加、叠加上限与顺序；折扣券与满减券能否同用，与会员价 / 其他平台优惠的优先级。
  • 券适用范围边界：单品 / 品类 / 全场；是否跨店。
  • App「我的优惠券」入口位置与样式：放在「我的订单」之后？或单独在「我的」列表新增一行？
  • 提交订单页选券交互：选中券后是否允许修改？多张券的推荐逻辑（默认选最大抵扣？）。
  • 直播发券后的通知方式：App/小程序 弹窗一次？消息中心持久化？PUSH？
  • 数据看板 ROI 口径：券带来的 GMV 增量如何归因。
  • 后续接任务引擎时，券作为任务奖励的发放形态（一次性发指定模板 / 用户自选）。
