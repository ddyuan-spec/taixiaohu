# 泰小虎画像业务库数据提取规则

## 一、数据表结构

### 1.1 健康档案数据表

| 类型 | 具体字段 | 字段类型 | 存储说明 |
|-----|---------|---------|---------|
| **体型数据** | 身高(CM) | 字符串 | 数值+单位，如"170" |
| | 体重(kg) | 字符串 | 数值+单位，如"65" |
| | BMI | 字符串 | 计算值，如"22.5" |
| **身体体征** | 血压(mmHg) | 字符串 | 格式"收缩压/舒张压"，如"120/80" |
| | 心率(次/分) | 字符串 | 数值，如"72" |
| | 呼吸(次/分) | 字符串 | 数值，如"16" |
| | 体温(℃) | 字符串 | 数值，如"36.5" |
| **睡眠数据** | 睡眠时长 | 字符串 | 时长，如"7小时30分" |
| | 起夜次数 | 字符串 | 次数，如"1" |
| **就医资料** | 病例 | 图片 | 图片文件存储 |
| | 检查报告单 | 图片 | 图片文件存储 |
| | 药物 | 图片 | 图片文件存储 |
| **用户信息** | 用户关系 | 枚举 | 本人/其他人 |
| | 昵称 | 字符串 | 用户自定义名称 |
| | 性别 | 枚举 | 男/女 |
| | 出生日期 | 年月日 | 日期格式 |
| | 是否默认咨询人 | 布尔 | 是/否 |

### 1.2 记忆库存储结构

```
记忆库 (Memory Store)
├── 用户ID (user_id)
├── 会话ID (session_id)
├── 对话记录 (dialogue_history)
│   ├── 用户输入
│   ├── AI回复
│   └── 时间戳
├── 提取的画像信息 (extracted_profile)
│   ├── 字段名
│   ├── 字段值
│   ├── 置信度
│   └── 来源对话轮次
└── 提取状态 (status)
    ├── pending (待确认)
    ├── confirmed (已确认)
    └── rejected (已拒绝)
```

### 1.3 业务缓存存储结构

```
业务缓存 (Business Cache) - 7天有效期
├── 用户ID (user_id)
├── 待更新档案列表 (pending_updates)
│   ├── 档案ID
│   ├── 字段名
│   ├── 旧值
│   ├── 新值
│   └── 提取来源
└── 过期时间 (expire_at)
```

## 二、提取规则

### 2.1 提取触发条件

| 触发场景 | 说明 |
|---------|------|
| **主动提及** | 用户主动说出健康相关数据，如"我身高170" |
| **症状描述** | 用户描述症状时附带体征数据，如"血压有点高，150/90" |
| **用药记录** | 用户提及正在服用的药物 |
| **就医历史** | 用户提及近期就医、检查情况 |
| **睡眠描述** | 用户描述睡眠质量、时长等 |

### 2.2 字段提取规则

#### 体型数据提取

| 字段 | 提取规则 | 示例 |
|-----|---------|------|
| 身高 | 匹配模式：数字+cm/厘米/CM | "我身高170cm" → 170 |
| 体重 | 匹配模式：数字+kg/公斤/千克 | "体重65kg" → 65 |
| BMI | 自动计算：体重(kg) / (身高(m))^2 | 身高170+体重65 → 22.5 |

#### 身体体征提取

| 字段 | 提取规则 | 示例 |
|-----|---------|------|
| 血压 | 匹配模式：数字/数字+mmHg | "血压120/80" → 120/80 |
| 心率 | 匹配模式：数字+次/分/心跳/心率 | "心率72次/分" → 72 |
| 呼吸 | 匹配模式：数字+次/分/呼吸 | "呼吸16次" → 16 |
| 体温 | 匹配模式：数字+度/℃ | "体温36.5度" → 36.5 |

#### 睡眠数据提取

| 字段 | 提取规则 | 示例 |
|-----|---------|------|
| 睡眠时长 | 匹配模式：数字+小时/时/h | "睡了7个小时" → 7小时 |
| 起夜次数 | 匹配模式：数字+次/回 | "起夜2次" → 2 |

#### 用户信息提取

| 字段 | 提取规则 | 示例 |
|-----|---------|------|
| 昵称 | 用户自我介绍或指代他人时使用 | "我帮我妈问问" → 昵称：妈 |
| 性别 | 直接说明或通过名字推断 | "我是女生" → 女 |
| 年龄/出生日期 | 直接说明年龄或出生年份 | "今年50岁" → 计算出生年份 |
| 用户关系 | 根据对话上下文判断 | "帮我老婆问" → 其他人 |

### 2.3 多用户（昵称）处理规则

```
对话中涉及多个人员时的处理流程：

1. 识别用户指代
   - 第一人称"我" → 当前登录用户
   - 第三人称"我妈/我爸/我老婆" → 创建/关联其他人员档案
   
2. 档案关联规则
   - 首次提及"其他人" → 创建新档案，标记"用户关系=其他人"
   - 再次提及同一昵称 → 关联已有档案
   - 提及"本人" → 关联默认咨询人档案

3. 昵称冲突处理
   - 同一用户下不同昵称 → 分别存储
   - 相同昵称不同特征 → 提示用户确认是否为同一人
```

## 三、提取流程

### 3.1 标准提取流程

```
用户对话
    ↓
[意图识别] 判断是否包含健康档案相关信息
    ↓
[信息提取] 从对话中提取关键字段
    ↓
[置信度评估] 评估提取信息的可信度 (0-1)
    ↓
[记忆库存储] 存储到记忆库，状态=pending
    ↓
[业务缓存] 写入业务缓存，设置7天过期
    ↓
等待客户端确认...
```

### 3.2 置信度评估标准

| 置信度等级 | 说明 | 处理策略 |
|-----------|------|---------|
| **≥0.9** | 用户明确直接陈述 | 自动提取，低优先级确认 |
| **0.7-0.9** | 用户提及但表述模糊 | 标准确认流程 |
| **0.5-0.7** | 间接提及或需要推断 | 高优先级确认，详细说明来源 |
| **<0.5** | 信息不完整或不确定 | 不提取，等待用户明确说明 |

### 3.3 提取后处理规则

```
提取完成后：

1. 去重检查
   - 同一字段在缓存期内已存在 → 更新值并标记为"更新"
   - 同一字段值相同 → 跳过，不重复提示

2. 冲突检测
   - 新值与旧值差异过大 → 标记"异常值"，加强确认
   - 如：身高从170变为160，可能输入错误

3. 关联推荐
   - 提取到相关字段时，推荐补充关联字段
   - 如：提取到"血压高" → 推荐记录"心率"、"用药情况"
```

## 四、客户端确认流程

### 4.1 确认触发时机

| 时机 | 说明 |
|-----|------|
| **会话开始时** | 检查业务缓存，有待确认项则提示 |
| **用户主动查看** | 用户点击"健康档案"时展示待确认项 |
| **定期提醒** | 每7天提醒一次（缓存过期前） |

### 4.2 确认弹窗内容规则

```
弹窗标题：健康档案更新提醒

内容结构：
1. 说明来源："根据您上次的对话，我们提取到以下健康信息"
2. 展示字段：字段名 + 提取值 + 置信度（可选）
3. 用户操作：
   - [确认更新] → 存入业务库，清除缓存
   - [暂不更新] → 保留缓存，下次再提示
   - [删除] → 清除缓存，不再提示
   - [修改] → 允许用户手动修正值

特殊情况：
- 多人员档案 → 按昵称分组展示
- 敏感信息（如疾病） → 增加隐私提示
- 图片类资料 → 展示缩略图，确认后转存
```

### 4.3 用户操作后的数据处理

| 用户操作 | 记忆库状态 | 业务缓存 | 业务库 |
|---------|-----------|---------|--------|
| 确认更新 | confirmed | 清除 | 写入新值 |
| 暂不更新 | pending | 保留 | 不变 |
| 删除 | rejected | 清除 | 不变 |
| 修改后确认 | confirmed | 清除 | 写入修改后的值 |

## 五、异常处理规则

### 5.1 提取失败处理

| 失败场景 | 处理策略 |
|---------|---------|
| 无法识别数值单位 | 存储原始文本，置信度降低，加强确认 |
| 数值超出正常范围 | 标记"异常值"，提示用户确认 |
| 多值冲突 | 取最新提及的值，或提示用户选择 |
| 图片识别失败 | 保留图片，提示用户手动填写 |

### 5.2 隐私与安全

```
1. 敏感信息加密存储
   - 病历、检查报告等图片加密存储
   - 血压、心率等数据脱敏处理

2. 用户授权
   - 首次提取需用户授权"允许记录健康信息"
   - 可随时在设置中关闭自动提取

3. 数据保留
   - 已拒绝的数据保留30天后自动删除
   - 业务缓存7天过期
   - 已确认的数据长期保留，用户可手动删除
```

## 六、API 接口规范

### 6.1 提取信息接口

```
POST /api/profile/extract
Request:
{
    "user_id": "string",
    "session_id": "string",
    "dialogue": [
        {"role": "user", "content": "我身高170cm"},
        {"role": "assistant", "content": "..."}
    ]
}

Response:
{
    "extracted_fields": [
        {
            "field_name": "height",
            "field_value": "170",
            "confidence": 0.95,
            "source": "user_dialogue_turn_1"
        }
    ],
    "pending_confirmation": true
}
```

### 6.2 确认更新接口

```
POST /api/profile/confirm
Request:
{
    "user_id": "string",
    "updates": [
        {
            "field_name": "height",
            "field_value": "170",
            "action": "confirm"  // confirm / reject / modify
        }
    ]
}

Response:
{
    "success": true,
    "updated_fields": ["height"],
    "rejected_fields": [],
    "next_prompt_time": null
}
```

### 6.3 获取待确认列表接口

```
GET /api/profile/pending?user_id=xxx

Response:
{
    "pending_items": [
        {
            "id": "string",
            "nickname": "本人",
            "field_name": "height",
            "field_value": "170",
            "extracted_at": "2024-01-01T00:00:00Z",
            "expire_at": "2024-01-08T00:00:00Z"
        }
    ],
    "total_count": 1
}
```

## 七、示例场景

### 场景1：用户主动提供信息

```
用户：我今年50岁，身高170，体重有点超标75kg

提取结果：
- 年龄：50岁 → 出生日期：1974年
- 身高：170cm
- 体重：75kg
- BMI：自动计算 = 25.95（超重）

置信度：0.95
处理：存入缓存，下次会话提示确认
```

### 场景2：多人员档案

```
用户：帮我妈问问，她血压有点高，150/90

提取结果：
- 用户关系：其他人
- 昵称：妈
- 血压：150/90 mmHg

处理：
1. 创建"妈"的档案（如不存在）
2. 关联到当前用户下
3. 标记"是否默认咨询人=否"
4. 存入缓存，提示确认时按昵称分组
```

### 场景3：信息更新

```
用户：上次说我体重75kg，最近减肥了，现在70kg

提取结果：
- 体重：70kg（新值）
- 旧值：75kg

处理：
1. 检测到与已有数据差异
2. 标记为"更新"而非"新增"
3. 弹窗提示："检测到体重变化：75kg → 70kg，是否更新？"
```
