PART 1 基础功能说明

1.1 功能概述

泰小虎客户端已规划"任务体系"（新手引导、每日任务等），每个任务在客户端有明确的完成判定逻辑。以"完善健康档案"为例，存在三种情形：

  • 情形 1：用户直接完成开屏引导的健康填写 → 已完成
  • 情形 2：用户中途跳过、零数据 → 未完成
  • 情形 3：用户中途跳过但填了部分数据 → 未完成，但需保留已填数据并在后续引导中带入默认值补全

问题：这些完成逻辑散落在客户端代码中，运营无法自助配置，且不同任务判定标准差异大，缺少统一抽象。

目标：在后台管理（任务配置模块）提供一套通用、可配置的"完成条件"定义能力，让运营无需改代码即可定义/调整任务的完成判定、状态流转与奖励发放。本 PRD 同时约定：每个任务的判定必须落地为埋点事件（见各任务的「埋点信息」章节，格式对齐《泰小虎埋点表 v2.3》）。

1.3 任务三态模型

采用显式三态模型，而非简单的是/否：

状态含义是否发虎力值客户端展示
未完成未触发任何完成/部分条件否"去完成"按钮（橙）
部分完成命中"部分完成"条件组（如跳过但填了部分）否（仅完成发）进度 + "继续完善"按钮（绿）
已完成命中"已完成"条件组是（发全额）"领取虎力值"（金）→ "已完成"灰
状态流转：未完成 → 部分完成 → 已完成 单向升级；一旦"已完成"且领取后不可逆。优先级解析：已完成 > 部分完成 > 未完成。

1.4 任务引擎配置模型

每个任务的"完成条件"由以下 6 个部分构成：

序号配置项说明控件
①判定类型当前统一为"埋点事件"（灰显固定，预留扩展）文本/禁用
②事件 Key监听的埋点事件名，如 profile_submit下拉 + 可输入
③属性条件组对事件属性做键值对判断，多行 AND 组合动态增删行
④状态映射将命中的条件组映射到 未完成 / 部分完成 / 已完成映射表
⑤续填策略开关 + 默认值来源（用于部分完成态的补全）开关
⑥奖励分配各态对应的虎力值（部分完成通常为 0）数字输入
设计取舍：判定数据源统一为"埋点事件"，"字段填充型"任务的子状态编码在事件属性里（如 status=completed），无需单独一类，模型保持干净。

1.5 属性条件组 · 配置规范

采用 键值对下拉 方式（运营友好，无需写代码）。每组条件由若干行组成，行内为：

[属性名]  [运算符]  [值]  （同一组内多行 = AND）

元素可选值示例
属性名该事件上报的属性字段status / filled / count / page
运算符等于 / 不等于 / 大于等于 / 小于等于 / 包含 / 非空存在== / >= / 非空
值字符串 / 数字 / 枚举completed / 1 / shop
  • 条件组 A → 已完成：如 status == completed
  • 条件组 B → 部分完成：如 status == skipped AND filled 非空
  • 均不命中 → 保持未完成

1.6 实时判定引擎

采用 实时事件驱动：埋点上报即刻判定，用户领奖无延迟。流程如下：

  1. 客户端 / 服务端上报埋点事件，必须携带 user_id 及判定所需属性（如 status、filled、count）。
  2. 后端事件流实时消费，取出该用户所有处于"未完成 / 部分完成"状态的任务配置。
  3. 对每个任务，依次用其"属性条件组"匹配当前事件：命中"已完成"组 → 置已完成并发奖；命中"部分完成"组 → 置部分完成（不发奖）；均不中 → 保持原态。
  4. 状态变更写回"用户任务进度表"，推送客户端刷新任务卡。
  5. 已完成态幂等：同一事件重复上报不重复发奖；部分完成可再次上报升级到已完成。

1.7 客户端协同

7.1 三态展示

  • 未完成：卡片显示"去完成"按钮（橙），点击跳对应页面。
  • 部分完成：卡片显示进度 + "继续完善"按钮（绿），点击进入带默认值的引导流。
  • 已完成（未领）：显示"领取虎力值"（金）→ 点击弹气泡、按钮变"已完成"灰不可点。

7.2 续填机制

续填本质是客户端行为，后台仅下发开关 + 默认值配置。客户端读取任务配置，进入引导流时：

preFill = 用户已填字段 ∪ 默认值补全项

例：用户跳过填了 gender/birth，下次进入引导这两个字段预填、其余用默认值，补完点完成 → 上报 status=completed → 任务升级为已完成、发奖。

1.8 埋点规范约束

  • 事件必须携带 user_id，否则无法关联用户任务进度。
  • 事件属性需包含判定所需字段（status / filled / count / page 等），缺失属性按"不命中"处理。
  • 实时上报（非批量延迟），保证判定时效。
  • 事件定义须登记于《泰小虎埋点表 v2.3》；已有事件直接复用，缺口事件（如 plan_create、checkin_record）须先新增埋点方可驱动经验。

1.9 边界与异常处理

异常场景处理策略
事件缺失判定属性按"不命中"处理，任务保持原态
同一事件多次上报已完成态幂等，不重复发奖
条件组为空该任务不可经事件完成（需人工 / 灰度兜底）
多条件组同时命中按优先级 已完成 > 部分完成 解析
部分完成 → 再次上报允许升级到已完成并发奖
