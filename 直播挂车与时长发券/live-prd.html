B. 项目背景与目标

业务背景（痛点分析）

在数字人能力上线之前，需先具备独立直播 H5 页面以承载直播带货场景。当前痛点：

  • 转化断点：直播场景下用户看到商品却缺乏顺畅的购买路径，直播间与商品交易域未打通，观看行为无法直接转化为下单。
  • 激励缺失：用户观看直播的时长未被沉淀为可运营的营销资产，缺少"看直播得福利"的正向激励，复看与停留时长偏低。
  • 能力割裂：直播底层能力（火山引擎）与泰小虎自有的商品、券体系未联动，商品卡与发券流程需由我方自研控制以统一体验。

具体场景：用户在 App/小程序看到一场"科学减重专场"直播，边看边对主播推荐的健康好物产生兴趣，但无法直接下单；看完直播也无任何后续触达，观看时长被白白浪费。

项目目标

业务目标

  • 打通"直播间 → 商品详情页 → 下单"转化链路，提升直播带货转化。
  • 将观看时长沉淀为可运营资产，通过发券激励提升用户停留与复看。
  • 以最小成本复用火山直播能力 + 优惠券体系，不自建发券引擎。

用户目标

  • 边看直播边买：在直播间直接看到商品卡并跳转购买。
  • 看直播得福利：观看达到时长即获得优惠券，感知明确。
  • 无打扰：未达时长、未登录期间不被打扰，券到账后登录自然感知。

C. 业务流程与架构

业务流程图（主流程 + 分支）

上图为主流程横向视图，菱形为判断节点，灰色分支为未登录/达限领的异常处理；主流程核心：进入 → 登录判断 → 观看 → 挂车点击 → 时长上报 → 用户离开 → T+1 批处理 → 限领判断 → 登录后通知 → 去使用 → 商品列表核销。

异常 / 分支流程

编号触发场景处理路径 / 规则
E1未登录进入直播直跳登录页；未登录期间不计时、不发券；登录成功回到观看页从头计时。
E2弱网 / 火山回调丢失因发券为 T+1 批处理（非实时），无需"断网即退出"的实时发券兜底；弱网/回调丢失仅导致该时段时长缺失（数据采集层通用问题），不影响 T+1 对已沉淀累计时长的判定。
E3中途退出再进同一 live_id 下观看时长从 0 重置；需单次连续观看达阈值才发券（不累计）。
E4发券 API 调用失败幂等重试（以触发记录为幂等键防重）；仍失败记日志供排查；不发券不通知。
E5达限领上限本场静默不再发券；已发张数来自触发记录本场计数。
E6券过期未用券过期属优惠券体系职责，本需求仅记录触发，不持有券实例状态。

信息架构（前端展示层 / 后端配置层 / 数据层 / 外部系统）

四层结构：前端展示层（直播 H5 + App/小程序感知）、后端配置层（直播间/主播/挂车/数据看板，发券规则复用 A4）、数据层（本需求持有的 4 张表）、外部/关联系统（火山引擎、优惠券体系、商品中心）。

状态机与状态流转规则

本需求仅持有「直播场次 / 观看记录 / 发券触发记录」三类实体；券实例状态在优惠券体系内。

实体 A：直播场次（live session）

起始状态触发动作 (Action)目标状态业务规则 / 前置条件
未开始运营创建场次并开播直播中火山侧创建直播（live_id 由火山 CreateActivityAPIV2 返回），H5 可进入观看；时长采集与发券仅直播中有效。
直播中运营结束直播已结束停止时长采集，不再发券；本场已触发记录保留。
已结束生成回放回放中回放不触发发券（时长发券仅直播中有效）。
直播中运营紧急中止已结束同"结束直播"；已发券不受影响。

实体 B：观看记录（watch record，per user per live）

起始状态触发动作 (Action)目标状态业务规则 / 前置条件
未观看用户进入且已登录观看中开始计时（每次从 0；中途退出重置）；未登录不进入此态。
观看中用户退出 / 切后台暂停(退出)停止计时；本次时长清零（E3 重置策略，不累计）。
暂停(退出)用户再次进入观看中重新从 0 计时（E3）。
观看中直播结束已结束(待结算)本场时长停止并沉淀，待 T+1 批处理结算。
已结束(待结算)T+1 批处理读累计时长已结算由发券触发记录（实体 C）承接：达阈值发券，未达标记"时长不足未发"。

实体 C：发券触发记录（issue_trigger_log，T+1 结算生成）

起始状态触发动作 (Action)目标状态业务规则 / 前置条件
未结算T+1 批处理：达阈值且本场未达限领已触发(成功)调优惠券体系发券 API 成功；以 user_id+live_id 为幂等键（单阈值无梯度维度）。
未结算T+1 批处理：达阈值但本场已达限领已上限(不发)静默跳过，不再发（用户无感知）。
未结算T+1 批处理：未达阈值未发(时长不足)本场累计时长不足，标记未发，本场不再结算。
未结算T+1 批处理：API 调用失败失败(待重试)幂等重试；仍失败记日志，不发券不通知。
已触发(成功)用户登录 App/小程序已通知登录成功第一个页面即弹窗 + 消息中心红点；同一批券只通知一次（去重）。
失败(待重试)重试成功已触发(成功)幂等保证，不重复发券。
已上限(不发)/未发—终态本场不再发券。

系统间数据流转图

数据流转：① 商品/挂车由我方后台同步到火山直播 → ② 火山下发直播流+挂车到用户端 → ③ 用户点击挂车跳转当前环境商品详情页（交易域旁路）→ ④ 火山 SDK 心跳上报观看时长到我方数据层 → ⑤ T+1 批处理按累计时长调优惠券体系发券 API → ⑥ 生成券实例进入用户券包 → ⑦ 用户登录 App/小程序触发弹窗+红点通知，去使用跳商品列表（闭环）。

D. 功能需求详情

D.1 需求影响范围

  • 不冲击老版本 / 老数据：本需求为独立分块，新增独立直播 H5 + 接入火山引擎 + 调用优惠券体系发券 API；不修改既有交易域、用户域核心逻辑。
  • 不新建发券引擎：发券规则（时长阈值、限领、有效期、券模板）全部复用优惠券体系 A4，本需求仅做触发与展示，不持有券实例/券模板/商品底价。
  • 新增数据表：直播场次表、观看记录表、发券触发记录、挂车关系表、主播表（均属本需求数据层）。
  • 影响现有功能：App/小程序"消息中心"需新增"券到账"记录类型与红点；登录后首屏需支持获券弹窗（不影响既有登录流程）。
  • 权限影响：平台后台新增"APP直播"菜单及「直播间管理」「主播管理」「挂车配置」「直播数据看板」子权限；运营角色需单独授权。

业务逻辑 / 计算规则

  • 文本截断规则：直播标题最多 17 字，超长用「…」截断；主播简介最多 30 字，超长用「…」截断；商品名称在卡片内最多 2 行，超出用「…」截断。
  • 价格显示：商品卡显示「直播价 ¥XX划线价 ¥XX」，划线价来自商品库，直播价 = 商品库当前活动价（无活动则取销售价）。
  • 观看时长计算：以火山观播 SDK 心跳回调为基准，每次心跳上报 duration_delta（秒）；本次观看时长 = Σ(duration_delta)，有效心跳需满足 2s ≤ 间隔 ≤ 120s。
  • 发券判定：单次连续观看时长 ≥ A4 配置的阈值（秒）时，该用户/本场次进入 T+1 发券候选集。
  • 中途退出重置：用户退出观看页或切后台 > 5s，视为本次观看结束，再次进入时长从 0 重新累计。
  • 挂车商品上限：单个直播间最多挂载 N 个商品，N 由后台「挂车上限」配置，默认 10。

弱网/火山回调丢失仅丢失该时段时长，不影响 T+1 对已沉淀时长的判定。

E. 验收标准

以下采用 Given（前提）→ When（操作）→ Then（结果）描述关键场景。

AC-1 未登录进入直播
Given 用户未登录进入直播 H5 ｜ When 页面加载完成 ｜ Then 直接跳转登录页，期间不采集观看时长、不发券；登录成功后回到观看页从头计时。
AC-2 单次连续达阈值发券
Given 用户已登录且某 live_id 正在直播 ｜ When 单次连续观看时长 ≥ A4 配置阈值 ｜ Then T+1 批处理静默调发券 API 成功，写触发记录（已触发），H5 无浮层。
AC-3 中途退出重置
Given 用户已观看 8 分钟（阈值 10 分钟） ｜ When 中途退出后再进入 ｜ Then 观看时长从 0 重置，需重新连续观看达阈值才发券（不累计）。
AC-4 达限领上限静默跳过
Given 该用户该 live_id 已达限领 N 张 ｜ When T+1 批处理判定 ｜ Then 静默跳过，不重复发券，无用户感知。
AC-5 登录后获券弹窗
Given 用户在前序场次已静默获券 ｜ When 用户登录 App/小程序成功进入第一个页面 ｜ Then 弹出"获得 X 张优惠券"弹窗 + 消息中心红点；同一批券仅通知一次。
AC-6 去使用跳转
Given 获券弹窗展示 ｜ When 用户点击"去使用" ｜ Then 直接跳转商品列表页（非我的优惠券）。
AC-7 后台创建直播间
Given 运营在后台新建直播并开播 ｜ When 提交 ｜ Then 后台调火山 CreateActivityAPIV2 返回 live_id，列表新增该直播间（状态=未开始/直播中），场次ID 由火山返回。
AC-8 挂车超上限
Given 某直播间已挂 N 个商品 ｜ When 运营尝试再挂第 N+1 个 ｜ Then 提示"本场最多挂 N 个商品"，保存失败。
AC-9 火山控制失败提示
Given 后台详情页点击"开播/停播" ｜ When 火山接口返回失败 ｜ Then 气泡提示「更改失败，失败原因【接口返回信息】」，状态不变。
AC-10 发券 API 失败重试
Given T+1 批处理调发券 API 失败 ｜ When 重试仍失败 ｜ Then 触发记录标记"失败(待重试)"并记日志，不发券不通知，幂等键防重。

G. 非功能性需求

维度要求
性能观看时长心跳上报轻量、可高并发写入数据层；T+1 批处理需支持全量观看记录扫描，设批处理窗口与失败重试；直播流低延迟播放（依赖火山）。
兼容性用户端支持微信小程序、App（iOS / Android）、独立 H5 浏览器；环境识别失败兜底 H5 商品详情页；后台适配主流桌面浏览器。
安全泰小虎 UserId 透传需防篡改（签名/校验）；观看时长防刷（心跳间隔异常丢弃）；发券幂等（user_id+live_id 幂等键）；后台操作需权限校验。
埋点要求关键转化链路（进入→观看→挂车点击→发券触发→通知→去使用）全埋点，参数含 user_id/live_id 以便归因。
可用性发券为 T+1 批处理，弱网/火山回调丢失不影响已沉淀时长判定，无需实时发券兜底。
