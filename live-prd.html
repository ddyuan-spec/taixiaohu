二、项目背景与目标

2.1 业务背景（痛点）

  • 现状：泰小虎已在直播后台（自有，底层封装火山引擎企业直播 OpenAPI）开播，直播间需要挂载「商品卡片」供观众点击下单；同时平台端「商品中心」已沉淀全量健康好物商品（含主图、名称、售价、划线价、卖点、H5 详情页路由等结构化字段）。
  • 痛点 1（重复录入）：目前若要在直播间上架商品卡片，需在直播后台逐个手动录入商品信息，与商品中心数据重复维护，效率低、易出错、易与商城真实信息不一致。
  • 痛点 2（无批量能力）：一场直播常需挂多个商品，逐条手填耗时；缺乏「一次勾选、批量推送」的运营工具。
  • 痛点 3（无状态可视）：运营无法在平台端直观看到「哪些商品已推送到直播间 / 推送失败」，缺乏统一操作台。

2.2 项目目标

  • 业务目标：打通「平台端商品中心 → 直播后台商品卡片」的自动化推送链路，让运营在平台端后台一处即可完成选品、批量推送、状态跟踪，减少直播上架的重复录入与出错。
  • 用户（运营）目标：在平台端后台新建「直播商品」页面，支持从商品中心选品入池、勾选后一键「同步到直播间」，字段全自动组装、无需任何手动填写；支持重复推送、失败重试、状态查看。

3.1 业务流程图（双泳道 · 横向 UML 风格）

主流程共 9 步（①开始 → ②商品中心 → ③添加入池 → ④直播商品列表页 → ⑤勾选/选直播间/同步 → ⑥后端组字段 → ⑦API调用链 → ⑧UpdateActivityProduct → ⑨回写状态 → 结束）；分支 A/B/C 独立于主线、互不影响。异常分支（推送失败）见 ⑨ 回写「同步失败」及 §4.2 批量失败处理。

3.2 需求本质与结论

本需求本质是平台端后台 → 直播后台（自有封装 OpenAPI）的商品卡片推送集成：将平台端「商品中心」已铺商品，经「直播商品列表」入池后，由运营手动勾选并推送到指定直播间的商品卡片。核心约束：

  • 入池 ≠ 推送：添加商品仅写入本地「直播商品列表」，不调任何 API、不影响直播间。
  • 推送是显式动作：运营在列表页勾选商品 → 打开「同步到直播间」弹窗 → 选目标直播间 → 确认推送，才经 OpenAPI 写入直播后台卡片。
  • 字段全自动：卡片 15 字段由后端按映射表自动组装（系统携带 5 / 固定传 1 共 2 / 不传默认 7 / 自动获取 1），运营无手动填写。
  • 允许重复推送：同商品可多次推送同一直播间（新增卡片，Id=0），受冷却防刷约束（1 分钟 3 次 / 冷却 3 分钟）。
  • 更新/删除解耦：直播商品列表的更新、移出，均不影响直播间已有卡片；反之亦然。

接口方法用途调用时机
ListActivityAPIPOST拉取直播间列表（分页/搜索/状态）选直播间、点「同步直播间信息」时

⚠️ 防误删关键约束
调用 UpdateActivityProduct 时，必须将不需要删除/更新的原有卡片配置原样完整传入，否则漏传的卡片会被删除或清空。因此推送前必须先 GetActivityProducts 拉全量卡片，再在其基础上追加新卡片（Id=0）。
技术约束

3.7 状态机与状态流转规则

核心实体：直播商品列表中的「单个商品行」的同步状态。状态按该商品最后一次推送的结果显示（直播间单选，一次只推 1 个直播间）。

起始状态触发动作（Action）前置/后置条件目标状态
—（仅入池，未推送过）从商品中心「添加到列表」入池成功后置：写入直播商品列表；推送状态显示「—」空值— 空值
已同步 / 同步失败点击「同步到直播间」并确认推送前置：已选目标直播间；所选商品无冷却中项（否则拦截报错）同步中（瞬时）→ 推送结果
同步中OpenAPI 返回成功后置：记录最后同步时间已同步
同步中OpenAPI 返回失败 / 冷却校验不通过后置：气泡提示失败原因；冷却失败提示：「xxx商品冷却中，推送失败请稍后再试」同步失败
已同步 / 同步失败单位时间内重复推送超阈值前置：1 分钟内同商品同直播间已推 3 次推送时拦截，不进入冷却状态，直接返回失败并提示「xxx商品冷却中，推送失败请稍后再试」
任意状态「移出」直播商品列表后置：不调 API，不影响直播间已有卡片（记录删除，退出状态机）
任意状态「更新」商品字段后置：仅更新列表字段，不改变同步状态、不自动推送（同步状态维持不变）
状态取值枚举
仅两种持久状态：已同步 / 同步失败。入池但尚未推送过直播间的商品，推送状态列显示「—」空值（非独立状态，仅表示尚无推送结果）。同步中为确认推送期间的按钮 loading 瞬时态（不落库、不在状态列展示）。无「待同步」「冷却中」状态——冷却为推送时校验拦截并直接返回失败（状态按最后一次结果覆盖）。

（二）业务逻辑与边界条件
【按钮】同步直播间（逐步 + 异常分支 + 校验文案）
  1. 前置校验：未选目标直播间 → 提示「请选择目标直播间」；已选商品为空 → 提示「请至少选择 1 个商品」。
  2. 冷却校验：逐商品校验「同商品+同直播间」冷却计数，冷却中的商品本次返回失败并提示「xxx商品冷却中，推送失败请稍后再试」；若全部冷却 → 提示「所选商品均在冷却中，请稍后再试」并中止，不调用 API。
  3. 字段组装：后端按 §三.3 字段映射表从《商品表》读取字段，组装 Products 数组元素（含 App H5 路由自动获取、EnableStatus=1、IsOrderMsgEnable=1）。
  4. API 调用链：
    1. 调 GetActivityProducts（GET）获取该直播间现有全部卡片（防误删关键）。
    2. 组装 UpdateActivityProduct（POST）请求：现有卡片原样保留 + 本次新增卡片（Id 传 0，系统分配新卡片 ID；允许同名）。
    3. 调用成功 → 成功商品回写「已同步」+ 最后同步时间。
  5. 批量失败处理：成功商品不在返回结果中；仅返回失败项 → 失败商品回写「同步失败」。前端统一 toast 汇总「成功 X 个，失败 Y 个」+ 失败商品名及原因（逐条列出失败项，非逐条气泡，避免刷屏）。
  6. 并发防重：请求进行中按钮 loading，禁止重复提交。

五、非功能性需求

5.1 性能

  • 批量推送单次支持 ≥ 20 个商品；后端遵守 UpdateActivityProduct 10 次/秒频率限制，必要时排队/合并。

5.2 兼容性

  • 平台端后台 Web，兼容 Chrome / Edge 最新两个大版本；分辨率 ≥ 1366×768。

5.3 安全

  • OpenAPI 调用凭证（AK/SK 或封装后凭证）由后端保管，不下发前端。
  • 操作按角色鉴权（直播运营）；推送/移出等敏感操作记录操作日志（操作人/时间/对象/结果）。

  • 本需求主体为平台端后台运营操作，不涉及《泰小虎埋点表 v2.3》所定义的用户端（Android / iOS / H5 / 微信小程序）行为埋点，故本期无新增用户端埋点事件。
  • 如需统计后台运营行为（推送成功/失败次数、冷却触发等），以后端既有操作日志方式记录即可，不额外接入独立日志服务、不占用用户端埋点体系。
  • 埋点规范总纲（如后续需扩展用户端事件时引用）：泰小虎埋点规范 v2.3（在线查阅）。新增事件须走埋点评审，禁止在本 PRD 内凭空定义事件英文名/属性变量名。

本期涉及埋点事件清单：无（本期不新增用户端埋点事件）。

六、验收标准

#场景（Given / When / Then）
AC1Given 运营在商品中心已铺商品，When 打开添加弹窗勾选商品点「添加到列表」，Then 商品写入直播商品列表且状态=「已同步」，不产生任何直播间调用。
AC2Given 添加弹窗中商品已存在于列表，When 查看该行，Then 该行置灰且复选框禁选，无法重复添加。
AC3Given 列表未勾选任何商品，When 点「同步到直播间」，Then 提示「请先勾选要推送的商品」，不打开弹窗。
AC4Given 勾选 2 个商品并选定 1 个直播间，When 点「同步直播间」，Then 后端先 GetActivityProducts 拉全量卡片再 UpdateActivityProduct 追加（Id=0）新卡片，成功商品状态→「已同步」并记录时间。
AC5Given 同一商品向同一直播间连续推送，When 第 2 次推送，Then 直播间新增一个同名独立卡片（非覆盖），验证允许重复推送。
AC6Given 同商品同直播间 1 分钟内已推 3 次，When 第 4 次推送该商品，Then 推送拦截返回失败，提示「xxx商品冷却中，推送失败请稍后再试」，该商品状态=「同步失败」。
AC7Given 批量推送 3 个商品其中 1 个失败，When 收到结果，Then 成功 2 个状态→「已同步」，失败 1 个状态→「同步失败」，并统一 toast 汇总「成功 2 个，失败 1 个」+ 该失败商品名及原因。
AC8Given 某商品状态为「同步失败」，When 筛选「同步失败」→勾选→选直播间→点同步，Then 可重新推送（无需独立重试按钮）。
AC9Given 已同步商品，When 在列表点「更新」，Then 仅刷新列表字段，不自动推送、同步状态不变。
AC10Given 已同步商品，When 点「移出」并确认，Then 列表删除该行，直播间已有卡片不受影响。
AC11Given 在直播后台侧编辑/删除某卡片，When 回到平台端列表，Then 平台端列表数据不受影响（解耦）。
AC12Given 目标直播间卡片已达 500，When 推送新卡片，Then 该商品置「同步失败」并提示「该直播间商品卡片已达上限（500）」。
AC13Given 新建了直播间，When 点「🔄 刷新直播间」，Then 下拉出现最新直播间；网络失败时 toast 提示并保留上次结果。

七、关键边界设计检查清单

✓检查项处理说明
☑网络与异常状态弱网/断网/500：推送超时视为失败→商品置「同步失败」+提示「网络异常，请重试」；刷新直播间失败保留上次缓存并 toast。
☑登录与游客态后台功能需登录 + 直播运营权限；无权限不展示菜单/按钮。
☑极限值与空状态空列表缺省图文案；商品名 100 字上限（Title）；卖点最多 3 个每点≤6字；直播间 500 卡片上限提示。
☑版本兼容性纯新增页面，无老数据迁移；不影响现有商品中心/直播后台既有能力。
☑逆向流移出列表/直播后台删卡片均已定义解耦规则（互不影响）。
☑高并发与防重推送进行中按钮 loading 防重复提交；冷却防刷（同商品同直播间 1 分钟 3 次/冷却 3 分钟）；防误删（先 Get 全量再 Update）。
